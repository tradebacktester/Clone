# Deployment Recommendation Report

**Engine:** Autonomous Research & Self-Evolution Laboratory v1.0.0  
**Date:** 2026-06-30

---

## What is a Deployment Recommendation?

A Deployment Recommendation is a structured, evidence-based proposal generated by the Research Engine when an experimental strategy version has passed all validation requirements and outperforms the current production version.

It is **not** an automatic deployment. It is a request for human review and approval.

---

## Recommendation Types

| Type | Condition | Description |
|------|-----------|-------------|
| `deploy` | Validation passed + SUPERIOR + p<0.05 | Strong evidence — recommend deployment |
| `continue_testing` | Validation passed + SUPERIOR or EQUIVALENT but not significant | Insufficient evidence — extend testing |
| `archive` | Validation failed or INFERIOR | No improvement case — discard experiment |
| `rollback` | Deployed version degrading | Emergency — recommend revert |

---

## Recommendation Contents

Every deployment recommendation includes:

### 1. Title
Clear, specific description identifying the experimental version and its relationship to production.

### 2. Summary
Concise narrative covering:
- What was improved
- Key performance metrics
- Statistical confidence level
- Recommended action

### 3. Code Change Summary
For each research code change artifact:
- Change type (refactor/optimization/filter/feature/scoring)
- Target module
- Lines added/removed
- Test results (tests, static analysis, security check, performance benchmark)
- Pseudo-code showing before/after configuration

### 4. Performance Summary
Side-by-side production vs experimental metrics:
- Win Rate, Avg R:R, Profit Factor, Max Drawdown, Sharpe, Total Return

### 5. Risk Assessment
Numbered list of specific risks:
- Statistical significance concerns
- Regressions on individual metrics
- Sample size limitations
- Validation confidence level

### 6. Statistical Significance
Two-proportion z-test p-value for win rate difference.

### 7. Confidence Score
0–100 score from the validation pipeline, combining all 10 stage scores.

### 8. Validation Evidence
Stage-by-stage pass/fail and score for all 10 validation stages.

### 9. Potential Drawbacks
Specific identified risks if the recommendation is approved.

### 10. Rollback Plan
6-step rollback procedure if the deployed version later degrades.

---

## Rollback Plan Template

```
1. Halt new signal generation from experimental version immediately.
2. Revert configuration to production version {parentVersion}.
3. Close any open positions opened under experimental rules (if applicable).
4. Restore all rule thresholds to production values.
5. Log rollback event to research history with timestamp and reason.
6. Begin post-mortem analysis of experimental version failure.
```

---

## Degradation Detection

After a version is approved and deployed, the research engine continues monitoring:

| Metric | Alert Threshold |
|--------|----------------|
| Win Rate drop | >8pp decline from deployed baseline |
| Profit Factor drop | >0.2 decline AND current PF <1.0 |

When degradation is detected:
1. Alert generated and logged to `rl_history`
2. Rollback recommendation created and queued
3. Operator notified via approval queue
4. **No automatic rollback** — human approval required
